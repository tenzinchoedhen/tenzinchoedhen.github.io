Real-Time Weather Data Pipeline

Apache Airflow for orchestration, dbt for transformation, Apache Superset for visualization, Docker for containerization

  • GitHub Repository

Introduction

This project implements a fully automated real-time weather data pipeline that fetches, transforms and visualizes weather conditions for New York City every 5 minutes. Built using modern data engineering tools and containerized with Docker, the pipeline demonstrates end-to-end data engineering capabilities from API extraction to interactive dashboards.

Tools and Technologies

  • Data Sources: Weatherstack API
  • Orchestration: Apache Airflow 3.0 for workflow automation and scheduling
  • Database: PostgreSQL 14 for data storage
  • Transformation: dbt 1.9 for SQL-based data modeling and transformation
  • Visualization: Apache Superset 3.0 for interactive dashboards
  • Containerization: Docker & Docker Compose for environment consistency
  • Update Frequency: 5-minute intervals for both data collection and dashboard refresh

Architecture Overview

Figure 1: Architecture Diagram of the Data Pipeline

The pipeline follows a modern ELT (Extract, Load, Transform) architecture with a layered data warehouse approach:

  1. Extract: Python scripts fetch real-time weather data from Weatherstack API every 5 minutes
  2. Load: Raw data is stored in the raw_weather_data table in PostgreSQL
  3. Transform: dbt models process the data through multiple layers:
    • Staging Layer: stg_weather_data - cleaned and standardized data
    • Mart Layer:
      • weather_report - comprehensive weather metrics
      • daily_average - aggregated daily statistics
  4. Report: Apache Superset queries the mart tables for fast visualization with 5-minute auto-refresh
  5. Orchestrate: Apache Airflow schedules and automates the entire pipeline
  6. Containerize: Docker ensures consistent deployment across environments

Key Insights

1. Pipeline Reliability at Scale

Running at 5-minute intervals generates 288 data points daily (12 per hour Ã— 24 hours). Over a month, this produces ~8,640 records, demonstrating the pipeline's capability to handle high-frequency data ingestion without performance degradation. Airflow's task monitoring showed 99%+ success rate with automatic retries handling transient API failures.

2. Staging Layer Value

The stg_weather_data staging layer proved essential for data quality. Approximately 2-3% of raw API responses contained null values or inconsistent formatting. The staging transformation catches these issues before they propagate to mart tables, ensuring dashboard accuracy.

3. Incremental vs Full Refresh Performance

Initial dbt models used full table refreshes, causing 8-12 second transformation times. Switching to incremental models reduced this to 1-2 seconds by processing only new records, critical for maintaining 5-minute SLAs as historical data grows.

4. Containerization Benefits

Docker containerization enabled seamless deployment across development and production environments. The entire stack (5 services: PostgreSQL, Airflow, dbt, Superset, Redis) starts with a single docker-compose up command, eliminating dependency conflicts and reducing setup time from hours to minutes.
